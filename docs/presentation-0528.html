Part 1

JWT 인증 시스템

Spring Security + JWT

왜 JWT를 선택했는가?


Session 방식

  • 서버에 세션 저장 → Scale-out 어려움
  • 서버 재시작 시 세션 유실
  • 서버 간 세션 공유 필요 (Sticky Session)
  • WebSocket 연결 시 세션 전달 복잡

JWT 방식 ✓

  • 서버 무상태(Stateless) → 수평 확장 용이
  • 토큰 자체에 사용자 정보 포함
  • 서버 간 공유할 것이 없음
  • HTTP & WebSocket 동일 방식으로 인증

Tech-Sync의 핵심 요구사항: 실시간 협업 에디터 → 다수의 WebSocket 연결 + REST API 동시 처리
→ Stateless + 토큰 기반 인증이 가장 적합

핵심 코드 ① JwtProvider

토큰 생성 · 검증 · 파싱을 담당하는 유틸리티 클래스

@Component
public class JwtProvider {

    private final SecretKey key;   // HMAC-SHA 서명 키
    private final long accessExpiration;   // 30분
    private final long refreshExpiration;  // 7일

    // ── 토큰 생성 ──
    private String buildToken(Long userId, String email, long expiration) {
        Date now = new Date();
        return Jwts.builder()
                .subject(String.valueOf(userId))  // 사용자 식별
                .claim("email", email)            // 커스텀 클레임
                .issuedAt(now)
                .expiration(new Date(now.getTime() + expiration))
                .signWith(key)                    // HMAC-SHA 서명
                .compact();
    }

    // ── 토큰 검증 ──
    public boolean isValid(String token) {
        try {
            parse(token);   // 서명 + 만료 동시 검증
            return true;
        } catch (JwtException | IllegalArgumentException e) {
            return false;
        }
    }
}

핵심 코드 ② JwtFilter

매 HTTP 요청마다 실행되는 인증 필터

@Component
public class JwtFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain chain) {
        // 1. Authorization 헤더에서 Bearer 토큰 추출
        String token = resolveToken(request);

        // 2. 토큰 유효성 검증
        if (StringUtils.hasText(token) && jwtProvider.isValid(token)) {

            // 3. 토큰에서 사용자 정보 추출
            Long userId = jwtProvider.getUserId(token);
            String email = jwtProvider.getEmail(token);

            // 4. SecurityContext에 인증 객체 세팅
            UsernamePasswordAuthenticationToken authentication =
                new UsernamePasswordAuthenticationToken(
                    userId, email, List.of());
            SecurityContextHolder.getContext()
                .setAuthentication(authentication);
        }

        chain.doFilter(request, response);  // 다음 필터로
    }
}

핵심 코드 ③ SecurityConfig

Spring Security 설정 — 세션 정책, 경로 인가, 필터 체인

@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http)
            throws Exception {
        http
            // CSRF 비활성화 (JWT 사용 시 불필요)
            .csrf(csrf -> csrf.disable())

            // 핵심: 세션을 생성하지 않음 → Stateless
            .sessionManagement(session ->
                session.sessionCreationPolicy(
                    SessionCreationPolicy.STATELESS))

            // 경로별 인가 규칙
            .authorizeHttpRequests(auth -> auth
                .requestMatchers(
                    "/api/auth/**",    // 인증 API (로그인/회원가입)
                    "/api/keywords",   // 키워드 조회
                    "/ws/**"           // WebSocket 엔드포인트
                ).permitAll()
                .anyRequest().authenticated()
            )

            // JwtFilter를 Spring 기본 필터 앞에 등록
            .addFilterBefore(jwtFilter,
                UsernamePasswordAuthenticationFilter.class);

        return http.build();
    }
}

// AuthServiceImpl — 로그인 처리
public AuthResponse login(LoginRequest request) {
    User user = userRepository.findByEmail(request.email())
        .orElseThrow(() -> new BusinessException("이메일 또는 비밀번호가 올바르지 않습니다."));

    if (!passwordEncoder.matches(request.password(), user.getPassword()))
        throw new BusinessException("이메일 또는 비밀번호가 올바르지 않습니다.");

    // Access Token + Refresh Token 발급
    String accessToken  = jwtProvider.createAccessToken(user.getUserId(), user.getEmail());
    String refreshToken = jwtProvider.createRefreshToken(user.getUserId(), user.getEmail());

    // Refresh Token → Redis 저장 (TTL 7일)
    redisTemplate.opsForValue().set(
        "refresh:" + user.getUserId(), refreshToken,
        Duration.ofMillis(jwtProvider.getRefreshExpiration()));

    return new AuthResponse(accessToken, refreshToken, user.getUserId(), user.getName());
}

  • 동시 다발 요청 시 1회만 refresh (Promise 공유)
  • Refresh도 실패 → 로그인 페이지로 리다이렉트
백엔드 (토큰 로테이션)
  • Refresh Token 유효성 + Redis 저장값 일치 확인
  • 새 Access + 새 Refresh 둘 다 발급
  • 이전 Refresh Token은 자동 무효화
  • → 탈취된 토큰 재사용 차단

설계 포인트 정리


설계 결정선택이유
세션 관리 STATELESS 서버에 상태 저장 X → 수평 확장 가능
Refresh Token 저장소 Redis 빠른 조회 + TTL 자동 만료
비밀번호 암호화 BCrypt 단방향 해시 + Salt → 레인보우 테이블 공격 방지
토큰 갱신 Rotation Refresh 시 새 토큰 쌍 발급 → 탈취 시 1회만 사용 가능
서명 알고리즘 HMAC-SHA 단일 서버 환경에서 단순하고 빠름

핵심: HTTP API와 동일한 JWT를 WebSocket(STOMP)에도 적용 → StompAuthChannelInterceptor가 CONNECT 시 토큰 검증

WebSocket에도 JWT 인증

HTTP의 JwtFilter와 같은 역할을 WebSocket에서 수행

@Component
public class StompAuthChannelInterceptor implements ChannelInterceptor {

    @Override
    public Message<?> preSend(Message<?> message, MessageChannel channel) {
        StompHeaderAccessor accessor = MessageHeaderAccessor
            .getAccessor(message, StompHeaderAccessor.class);

        // STOMP CONNECT 시에만 인증 수행
        if (!StompCommand.CONNECT.equals(accessor.getCommand()))
            return message;

        // Authorization 헤더에서 JWT 추출 → 검증
        String token = resolveToken(
            accessor.getFirstNativeHeader("Authorization"));
        if (token == null || !jwtProvider.isValid(token))
            throw new MessagingException("STOMP 인증에 실패했습니다.");

        // 인증 성공 → Principal 세팅
        Long userId = jwtProvider.getUserId(token);
        accessor.setUser(new UsernamePasswordAuthenticationToken(
            userId, email, List.of()));
        return message;
    }
}

동시 편집 충돌 해결 — OT

Operational Transformation으로 인덱스 충돌 보정


문제 상황

User A: 5번 위치에 "가" 삽입
User B: 3번 위치에 "나" 삽입

→ B의 삽입 후 A의 "5번"은 실제로 "6번"이 되어야 함
OT 해결

1. 서버: Redis로 seqNo 채번 → 순서 보장
2. 클라이언트: pending ops와 remote ops를 transform
3. Tie-break: userId가 작은 쪽 우선

→ 모든 클라이언트가 동일한 최종 상태에 수렴 

// 클라이언트 OT 핵심 — useEditorSocket.js
let remoteDelta = new Delta(broadcast.ops);
for (let i = 0; i < pending.length; i++) {
    const newRemote = pending[i].transform(remoteDelta, selfHasPriority);
    pending[i] = remoteDelta.transform(pending[i], !selfHasPriority);
    remoteDelta = newRemote;
}
onRemoteDelta({ ...broadcast, ops: remoteDelta.ops });
