# 실행 계획 (Execution Plans)

이 문서는 에이전트가 복잡한 작업을 수행할 때 따라야 하는 실행 계획을 정의한다.
**모든 복잡한 작업은 "즉시 코딩"이 아니라 "계획 → 승인 → 실행 → 검증" 순서로 진행한다.**

---

## 실행 계획이 필요한 경우

아래 조건 중 하나라도 해당하면, 코드 작성 전에 반드시 실행 계획을 먼저 제시하고 사용자 승인을 받는다.

- 2개 이상의 파일을 생성하거나 수정해야 하는 작업
- 새로운 API 엔드포인트를 추가하는 작업
- DB 스키마(Entity/Document) 변경이 포함된 작업
- WebSocket/실시간 로직을 수정하는 작업
- 기존 코드의 동작 방식을 변경하는 리팩토링

아래의 경우는 실행 계획 없이 바로 진행해도 된다.
- 단일 파일의 버그 수정
- 주석/문서 수정
- 단순 설정값 변경
- 사용자가 명시적으로 "바로 해줘"라고 요청한 경우

---

## 실행 계획 작성 형식

작업을 시작하기 전에 아래 형식으로 계획을 제시한다.

```
## 실행 계획: [작업 제목]

### 목표
[이 작업이 완료되면 어떤 상태가 되는지 한 문장으로]

### 영향 범위
- 생성 파일: [새로 만들 파일 목록]
- 수정 파일: [수정할 기존 파일 목록]
- 의존성: [새로 추가할 라이브러리가 있으면 명시]

### 실행 단계
1. [단계 1 설명]
2. [단계 2 설명]
3. ...

### 검증 방법
- [어떻게 이 작업이 올바르게 완료되었는지 확인하는가]

### 위험 요소
- [이 작업에서 주의해야 할 점, 기존 기능에 영향을 줄 수 있는 부분]
```

사용자가 계획을 승인하면 실행을 시작하고, 수정 요청이 있으면 계획을 수정하여 다시 제시한다.

---

## 작업 유형별 실행 계획 템플릿

### 1. 새 API 엔드포인트 추가

```
### 목표
[리소스명] CRUD API를 추가하여 [기능 설명]을 제공한다.

### 영향 범위
- 생성: *Request.java, *Response.java, *Controller.java, *Service.java, *ServiceImpl.java, *Repository.java
- 수정: SecurityConfig.java (경로 허용 설정이 필요한 경우)
- Entity가 없으면: *Entity.java 또는 *Document.java도 생성

### 실행 단계
1. DTO 작성 — Request/Response record 정의 (Validation 포함)
2. Entity/Document 작성 — 테이블/컬렉션 매핑
3. Repository 작성 — JPA/MongoRepository 상속, 커스텀 쿼리 필요 시 메서드 추가
4. Service 인터페이스 + 구현체 작성 — 비즈니스 로직
5. Controller 작성 — REST 매핑, ApiResponse 래퍼 사용
6. SecurityConfig 확인 — 새 경로의 인증 요구 수준 설정
7. 테스트 작성 — Service 테스트 (정상 1 + 실패 1 이상)
8. 컴파일 확인 — ./gradlew compileJava

### 검증 방법
- 컴파일 성공
- 테스트 통과
- 보안 체크리스트 점검 완료

### 위험 요소
- 새 Entity의 ddl-auto=update가 기존 테이블에 영향을 줄 수 있음
- SecurityConfig 변경 시 기존 인증 경로에 영향 없는지 확인
```

### 2. 새 Entity / MongoDB Document 추가

```
### 목표
[테이블/컬렉션명]을 추가하여 [데이터]를 관리한다.

### 영향 범위
- 생성: domain/*Entity.java (또는 *Document.java), repository/*Repository.java
- 수정: 없음 (또는 기존 Entity에 관계 추가 시 해당 Entity)

### 실행 단계
1. 스키마 설계 — 컬럼/필드 목록, 제약조건, 인덱스 정의
2. Entity/Document 클래스 작성
   - MariaDB: @Entity, @Table, Auditing 적용
   - MongoDB: @Document, @Id는 String
3. Repository 인터페이스 작성
4. 기존 Entity와 관계가 있으면 관계 매핑 추가 (단, 기존 Entity 수정은 승인 후)
5. 컴파일 확인

### 검증 방법
- 컴파일 성공
- ddl-auto=update로 로컬 DB에 테이블 생성 확인 (MariaDB)
- docs/domain.md에 새 테이블/컬렉션 정보 추가 제안

### 위험 요소
- 컬럼명이 기존 테이블과 충돌하지 않는지
- MongoDB 인덱스 설계가 실제 쿼리 패턴과 일치하는지
```

### 3. WebSocket / 실시간 기능 수정

```
### 목표
[기능 설명] — 실시간 협업 로직을 수정/추가한다.

### 영향 범위
- 수정: EditorService, WebSocket Controller, (필요 시) DELTA_LOG 관련 코드
- 프론트엔드: useSocket.js, EditorPanel 등에 변경이 필요한 경우 명시

### 실행 단계
1. 현재 로직 분석 — 기존 Delta 처리 흐름을 파악하고 요약
2. 변경 설계 — 어떤 부분을 어떻게 바꿀지 설명
3. [사용자 승인 대기]
4. 백엔드 수정 — Service/Controller 변경
5. DELTA_LOG 저장 로직 확인 — 변경으로 인해 로그 누락이 없는지
6. 프론트엔드 수정 (필요 시)
7. 검증

### 검증 방법
- DELTA_LOG에 새 Delta가 정상 저장되는지
- seqNo 채번이 원자적인지
- 브로드캐스트 순서가 올바른지
- 기존 자동저장(30초) 로직에 영향 없는지

### 위험 요소
- OT 변환 로직 변경은 모든 클라이언트에 영향 → 충분한 테스트 필요
- DELTA_LOG 스키마 변경 시 기존 데이터 마이그레이션 고려
- 브로드캐스트 순서가 바뀌면 클라이언트 간 문서 불일치 발생 가능
```

### 4. 알림 시스템 (SSE / Web Push) 수정

```
### 목표
[기능 설명] — 알림 발행/수신 로직을 수정/추가한다.

### 영향 범위
- 백엔드: AlarmService, SSE Controller, (필요 시) Redis Pub/Sub 설정
- 프론트엔드: useSSE.js, 알림 UI 컴포넌트

### 실행 단계
1. 현재 알림 플로우 분석 (이벤트 → Redis Pub/Sub → SSE or Web Push)
2. 변경 설계
3. [사용자 승인 대기]
4. 백엔드 수정
5. 프론트엔드 수정 (필요 시)
6. 검증

### 검증 방법
- Redis Pub/Sub 메시지가 정상 발행되는지
- SSE 연결된 클라이언트가 알림을 수신하는지
- 미연결 클라이언트에게 Web Push가 전송되는지 (해당 시)

### 위험 요소
- SSE 연결 수 증가 시 서버 리소스 영향
- Redis Pub/Sub 채널 네이밍 변경 시 기존 구독자 영향
```

### 5. 보안 / 인증 관련 변경

```
### 목표
[기능 설명] — 인증/인가 로직을 수정한다.

### 영향 범위
- 수정: SecurityConfig, JwtFilter, JwtProvider, (필요 시) AuthService

### 실행 단계
1. 현재 보안 설정 분석
2. 변경 설계 — 어떤 경로에 어떤 수준의 인증을 적용할지
3. [사용자 승인 대기 — 보안 변경은 반드시 승인 필수]
4. 코드 수정
5. 보안 체크리스트 전체 재점검
6. 기존 인증 테스트가 깨지지 않는지 확인

### 검증 방법
- 보호된 경로에 토큰 없이 접근 시 401 반환
- 유효한 토큰으로 접근 시 정상 응답
- Refresh Token 갱신 플로우 정상 동작
- 기존 permitAll 경로(/api/auth/**, /api/keywords, /ws/**)가 여전히 동작

### 위험 요소
- SecurityConfig 변경은 모든 API에 영향 → 변경 최소화 원칙
- JWT 키 관련 변경 시 기존 토큰 무효화 가능성
```

## 실행 중 규칙

### 단계별 보고
- 각 실행 단계 완료 후 간단한 진행 상황을 보고한다
- "2/5 완료: Repository 작성 완료. 다음: Service 구현"

### 문제 발생 시
- 계획에 없던 파일 수정이 필요해지면 → 즉시 멈추고 사용자에게 알린다
- 기존 코드에 예상치 못한 구조가 발견되면 → 계획을 수정하여 다시 제시한다
- 컴파일 에러가 발생하면 → 원인을 분석하고 수정 방안을 제안한 뒤 승인을 받는다

### 완료 보고
작업 완료 후 아래 내용을 요약한다.
```
## 완료 보고: [작업 제목]
- 생성한 파일: [목록]
- 수정한 파일: [목록]
- 테스트 결과: [통과/실패 여부]
- 보안 점검: [완료/해당없음]
- 주의 사항: [있으면 명시]
```
